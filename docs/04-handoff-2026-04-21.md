## What the data is (and isn't)

**Is**: a faithful, structured extraction of the MCG 26A PDF's event catalog. Every event declaration, prereq citation, applicability tag, and bracket annotation has been parsed into a consistent schema. Includes new metadata V1 lacked: structured `nameTag` arrays, `seriesInfo`, `forDownstream`/`datagroup` conditions, and `oneOf` prereq groups.

**Isn't**: a DAG. Per-event prereq lists are preserved as the MCG wrote them (including range-notation series members each carrying the same prereqs). The graph-build step — applying Rule G tight-serial collapse, per-student edge filtering (Rules A + I), oneOf rendering, condition-sentinel resolution — is the UI team's responsibility.

## How to consume `MCG-26A_2026-04-21.json`

Top-level keys:

```text
sourceDocument      "MCG 26A"
classVersion        "26A"
generatedDate       "2026-04-21"
goalEvents          [ 11 codes — see KB §5 ]
eventSchema         inline docs for every event field
prereqSchema        inline docs for every prereq field
conditionSchema     forDownstream | datagroup
prereqGroupSchema   id | kind | audienceHint | description
phases              [ { phase, phaseName, sourcePages, eventCount } × 9 ]
events              { "PF 8332F": {...}, ... }   ← 667 entries keyed by event code
counts              totals
```

**Every event has the same top-level keys.** Every prereq has the same keys. No conditional shapes; nullable-where-unused.

### Minimum viable event traversal

```js
const mcg = require('./MCG-26A_2026-04-21.json');
const goal = mcg.events['PF 8332F'];

for (const prereq of goal.prerequisites) {
  // prereq.code → next node to visit
  // prereq.nameTag → array of audience codes (or null)
  // prereq.nameTagRole → "crew-solo" | "non-crew-solo" | "select P" | "select" | "optional" | null
  // prereq.nameTagAircraft → "C-12" | "F-16" | "T-38" | "Learjet" | null
  // prereq.condition → null | {type: "forDownstream", target} | {type: "datagroup", values}
  // prereq.group → null | {id, kind: "oneOf", audienceHint, description}
}
```

### Rule A (applicability filter) — pseudo-code

```
keepEdge(viewer, mainEvent, prereq):
  if prereq.nameTag is null: return true
  return viewer.studentType ∈ (mainEvent.applicability ∩ prereq.nameTag)
```

See `mcg-knowledge-base.md` §6 for Rules A, B, C, D, E, G, H, I, J, K, L — all required for correct per-student DAG behavior.

### oneOf groups — handling "pick one of these"

5 oneOf groups in this dataset (see `extraction-summary.md` for the full list). Members of a group share `group.id`.

Rendering guidance: when traversing a prereq list, collapse all prereqs with the same `group.id` into a single "picks one" branch. A student is unlocked past the group if they've completed **any one** of the members (not all).

Example: SY 7511F's flight prereqs contain 3 alternatives with `group.id: "SY-7511F-flight"`. A pilot unlocks SY 7511F by completing any one of SY 7212F / SY 7222F / SY 7304F.

## Known open items for the UI team

### Schema gaps not encoded (need decisions at graph-build time)

| Case | Examples | Current encoding | What the UI/graph-build owns |
|---|---|---|---|
| `forDownstream` targets that aren't event codes | `target: "all"`, `target: "P"`, `target: "US Only Control Rooms with Special Access Programs"` | preserved as strings on `condition.target` | Decide: treat as unconditional (conservative), drop the edge (aggressive), or add typed condition variants |
| Timing constraints | `[Hour 4]` on FQ 6201A/8101A citations; `+ 5 days` between PF 8202E and PF 8211F | kept verbatim inside prereq `name` text; no structured field | Scheduling domain — the DAG doesn't need to know, but the Big Board rendering might |
| Observer audiences (paired events) | TF 6231C `(ABM/FTE execute, CSO/P/RPA observe)` | only executers in `applicability` | Per user ruling: **DAG should NOT display to observers**. Current encoding is correct. |

### Remaining MCG source drift (preserved verbatim; not fixed at data layer)

Per user ruling, these are MCG authoring errors. Preserved in `eventName` text so the data matches the PDF. Flag these back to the TPS curriculum office if you see them surface in the UI:

- **FQ 6321R** — suffix `R` (Written Report) but event name says "Oral Report"
- **PF 8330M TF 6251F prereq** — `[req for PF 8332F]` (missing apostrophe/d — unique in 26A)
- **PF 8230M / PF 8231F description** — copy-pasted from PF 8222F / PF 8221C (wrong topic)
- **TF 6240M description** — ends mid-sentence: `"...FQ 6241F C-12 S"`
- **SY 7504Y** tag in name — missing comma vs sibling events

### Single remaining dangling prereq

- **TF 5320A** lists `TF 5301A Introduction to Space Systems Testing (STC)` as a prereq. No event with this code exists in MCG 26A. Likely a 25A legacy that was never removed from TF 5320A's prereq list. Per user ruling, left verbatim. Graph-build may drop the edge or display the node with an "unresolved" marker.

### Condition type vocabulary

Two condition types are populated in the data:

```js
{ type: "forDownstream", target: "PF 7111C" }   // "required only if student's chain passes through PF 7111C"
{ type: "datagroup", values: ["T-38"] }          // "required only for students whose DG is in values"
```

See KB §7 for semantics. DG values seen: `"C-12"`, `"F-16"`, `"T-38"`, `"Learjet"`.

### Role vocabulary

Non-null `role` values in `applicabilityDetail.role`:

| Role | Count | Meaning |
|---|---|---|
| `crew-solo` | 12 | KB Rule D — DG-matched pilots only (paired with `aircraft`) |
| `non-crew-solo` | 4 | KB Rule E — complement of crew-solo |
| `select P` | 6 | KB §4 — pilots at instructor discretion |
| `select` | 2 | select subset of the audience (e.g. select STC) |
| `optional` | 2 | TF 5520F, TF 6232F — scheduling-level optional; student IS applicable but completion is discretionary |
